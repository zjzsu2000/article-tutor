# v0.2 — TTS read-aloud and quiz data structure

Date: 2026-05-16

This release focuses on listening support and lays groundwork for the
post-reading quest UI planned for v0.3.

## Highlights

- Text-to-speech read-aloud for the full article, a single sentence, or a
  single word, via the Web Speech API.
- Pause / Resume / Stop controls for full-article playback.
- Friendly fallback notice on browsers that don't expose Web Speech.
- Cleaner, more reliable switching between word / sentence / article
  speech.
- Per-article quiz data structure ready for the upcoming Challenge UI
  (闯关测试).

---

## TTS improvements

### Read-aloud surfaces

| Surface | Where it appears | What it reads |
|---|---|---|
| 🔊 Read aloud | Article header | The full article, sentence by sentence |
| 🔊 Read sentence | Sentence info panel | The selected sentence |
| 🔊 Read word | Word info panel | The selected word |

While the full article is playing, the current sentence is highlighted
so a young learner can follow along.

### Slower default rate

Default speech rate is now `0.7` (was `0.9`, briefly `0.8`). Real-device
feedback from young learners suggested anything faster felt rushed and
hard to follow. Pitch and volume stay at their natural defaults (`1.0`
each). A future minimal settings UI is planned to let the learner pick
Slow / Normal / Fast.

### Voice selection

The module prefers an `en-US` voice when available, falling back to any
`en-*` voice, then the system default. Voice loading is async on Chrome,
so the picker re-runs when the browser fires `voiceschanged`.

## Browser compatibility fallback

Some China / HarmonyOS tablet browsers do not expose
`window.speechSynthesis`, so the read-aloud buttons could not work
there. Previously the buttons were silently hidden — users couldn't tell
whether the feature was broken or simply missing.

The article header now renders a friendly notice in place of the button:

- zh: 当前浏览器暂不支持朗读功能，可以换其他浏览器或设备试试。阅读、点词和点句子功能仍可正常使用。
- en: Speech is not supported in this browser. You can try another browser or device. Reading, word lookup, and sentence explanation still work.

Reading, word lookup, and sentence explanation are not affected — they
work on the same browsers regardless.

## Pause / Resume / Stop

The article header primary control now follows the speech state:

- **Idle**: 🔊 Read aloud
- **Playing**: ⏸ Pause + ⏹ Stop
- **Paused**: ▶ Resume + ⏹ Stop

Pause / Resume are best-effort: they wrap `speechSynthesis.pause()` and
`resume()` inside `try / catch`. Browsers with unreliable pause behavior
degrade silently — a learner can still hit Stop and restart from the
beginning.

The per-sentence and per-word speak buttons stay as single-tap "fire and
forget" — utterances at that scale are short enough that pause is not
useful.

## Robust switching between word / sentence / article speech

Three switching glitches showed up in real use and are addressed in this
release:

1. **Stuck `speaking` state.** Tapping a new utterance before the
   previous one finished could leave the engine in a state where no
   `onend` ever fired, freezing the UI.
2. **Cancel-then-speak race on Chromium-family browsers.** Calling
   `speak()` immediately after `cancel()` could silently drop the new
   utterance.
3. **Highlight residue.** When a word / sentence tap took over from the
   full-article reader, the last highlighted sentence stayed highlighted
   for the rest of the new utterance.

Fixes:

- A `startFresh(queueFn)` wrapper checks whether the engine was active,
  calls `resume()` if it was paused, calls `cancel()`, and waits a short
  settle delay before queueing the new utterance. This is a common
  practical workaround for Chromium-family `speechSynthesis` timing
  issues.
- The TTS module exposes a `sequenceId` that bumps on every speak / stop.
  The article reader compares the global `sequenceId` to the one it
  owns; when they differ, the highlight clears immediately.
- All in-flight callbacks (`onstart` / `onend` / `onerror`) check their
  captured id against the current `sequenceId` and no-op if stale, so
  stale events from a cancelled utterance never poison state.

End result: tapping word → sentence → article in any order plays the
latest one, leaves no stuck buttons, and clears the highlight when
ownership changes.

## Quiz data structure added

Each article in `lib/mockData.ts` now carries an optional `quiz` field:

```ts
type QuestionType = "vocabulary" | "detail" | "main_idea";

type Question = {
  id: string;
  type: QuestionType;
  question: string;
  options: string[];      // exactly 4
  correctAnswer: string;  // must equal one of options
  explanation: string;
};

type Article = {
  // ...existing fields...
  quiz?: Question[];      // exactly 3 questions: vocab + detail + main_idea
};
```

All eight seed articles now have three questions each (one of each type).
Every `correctAnswer` is verified to match one of its `options` exactly.

The UI on top of this data is **not** in v0.2. It is the headline feature
of v0.3 (Challenge UI / 闯关测试).

## Deployment notes

The push-based static host attached to this repo (Vercel) builds on
every push to `main` automatically.

A second static host (e.g. Cloudflare Pages) **still requires manual
deployment** when it was set up in Direct Upload mode rather than via a
Git connection. From a local checkout that mirror can be updated with:

```bash
npm run build && \
  npx wrangler pages deploy out --project-name english-study --branch main
```

See [`docs/development/DEPLOYMENT.md`](../development/DEPLOYMENT.md) for
full deployment details and the credential-handling rules.
